# データ設計（SQLite）

**前提**
- 用語は Backlog 準拠で「課題（チケット）」「子課題（タスク）」「マイルストーン」を使用する。
- WBS 階層は「プロジェクト → 課題（チケット） → 子課題（タスク）」の2段とする。
- SQLite を前提に、日付は `YYYY-MM-DD` もしくは `YYYY-MM-DD HH:MM:SS` の TEXT で保持する。
- 真偽値は INTEGER の `0/1` で保持する。

**テーブル一覧**
- `users`: ユーザー
- `teams`: チーム
- `team_members`: チームメンバーと権限
- `projects`: プロジェクト
- `milestones`: マイルストーン
- `issues`: 課題（チケット）と子課題（タスク）
- `tags`: タグ
- `issue_tags`: 課題タグの中間

**テーブル定義**

**users**
用途: ログイン用ユーザー。個人情報は持たない。
主キー: `id`
外部キー: なし
インデックス: `users(username)` UNIQUE
カラム定義:
|列|型|NULL|説明|
|---|---|---|---|
|id|INTEGER|NO|ユーザーID|
|username|TEXT|NO|ユーザー名。ユニーク|
|password_hash|TEXT|NO|パスワードハッシュ|
|created_at|TEXT|NO|作成日|
|updated_at|TEXT|NO|更新日|

**teams**
用途: チーム。
主キー: `id`
外部キー: なし
インデックス: `teams(name)` UNIQUE
カラム定義:
|列|型|NULL|説明|
|---|---|---|---|
|id|INTEGER|NO|チームID|
|name|TEXT|NO|チーム名。ユニーク|
|created_at|TEXT|NO|作成日|
|updated_at|TEXT|NO|更新日|

**team_members**
用途: チームとユーザーの所属と権限。
主キー: `(team_id, user_id)`
外部キー: `team_id → teams.id`, `user_id → users.id`
インデックス: `team_members(user_id)`
カラム定義:
|列|型|NULL|説明|
|---|---|---|---|
|team_id|INTEGER|NO|チームID|
|user_id|INTEGER|NO|ユーザーID|
|role|TEXT|NO|`admin` / `member`|
|joined_at|TEXT|NO|参加日|
|updated_at|TEXT|NO|更新日|

**projects**
用途: チーム配下のプロジェクト。
主キー: `id`
外部キー: `team_id → teams.id`
インデックス: `projects(team_id)`, `projects(team_id, name)` UNIQUE
カラム定義:
|列|型|NULL|説明|
|---|---|---|---|
|id|INTEGER|NO|プロジェクトID|
|team_id|INTEGER|NO|所属チームID|
|name|TEXT|NO|プロジェクト名|
|description|TEXT|YES|説明|
|created_at|TEXT|NO|作成日|
|updated_at|TEXT|NO|更新日|

**milestones**
用途: 期間の区切り。
主キー: `id`
外部キー: `project_id → projects.id`
インデックス: `milestones(project_id)`, `milestones(project_id, display_order)`, `milestones(project_id, planned_start, planned_end)`
カラム定義:
|列|型|NULL|説明|
|---|---|---|---|
|id|INTEGER|NO|マイルストーンID|
|project_id|INTEGER|NO|プロジェクトID|
|name|TEXT|NO|名称|
|planned_start|TEXT|YES|予定開始日|
|planned_end|TEXT|YES|予定終了日|
|display_order|INTEGER|NO|表示順|
|created_at|TEXT|NO|作成日|
|updated_at|TEXT|NO|更新日|

**issues**
用途: 課題（チケット）と子課題（タスク）。`parent_issue_id` が NULL の場合は課題、NULL でない場合は子課題。
主キー: `id`
外部キー: `project_id → projects.id`, `parent_issue_id → issues.id`, `milestone_id → milestones.id`, `assignee_user_id → users.id`
インデックス: `issues(project_id, parent_issue_id)`, `issues(project_id, milestone_id)`, `issues(project_id, status)`, `issues(project_id, priority)`, `issues(project_id, planned_start, planned_end)`, `issues(assignee_user_id)`
カラム定義:
|列|型|NULL|説明|
|---|---|---|---|
|id|INTEGER|NO|課題ID|
|project_id|INTEGER|NO|プロジェクトID|
|parent_issue_id|INTEGER|YES|親課題ID。子課題のときに設定|
|milestone_id|INTEGER|YES|所属マイルストーンID|
|title|TEXT|NO|タイトル|
|status|TEXT|NO|状態。例: `未着手` / `進行中` / `完了`|
|planned_start|TEXT|YES|予定開始日|
|planned_end|TEXT|YES|予定終了日|
|planned_minutes|INTEGER|YES|予定時間（分）|
|actual_minutes|INTEGER|YES|実績時間（分）|
|priority|INTEGER|YES|優先度。小さいほど高い想定|
|assignee_user_id|INTEGER|YES|担当者ユーザーID|
|daily_reset_flag|INTEGER|NO|デイリータスクフラグ（0/1）|
|completed_at|TEXT|YES|完了日時。週次実績集計に使用|
|created_at|TEXT|NO|作成日|
|updated_at|TEXT|NO|更新日|

**tags**
用途: タグと色。
主キー: `id`
外部キー: `team_id → teams.id`
インデックス: `tags(team_id)`, `tags(team_id, name)` UNIQUE
カラム定義:
|列|型|NULL|説明|
|---|---|---|---|
|id|INTEGER|NO|タグID|
|team_id|INTEGER|NO|所属チームID|
|name|TEXT|NO|タグ名|
|color|TEXT|YES|色（例: `#FF9900`）|
|created_at|TEXT|NO|作成日|
|updated_at|TEXT|NO|更新日|

**issue_tags**
用途: 課題とタグの中間。
主キー: `(issue_id, tag_id)`
外部キー: `issue_id → issues.id`, `tag_id → tags.id`
インデックス: `issue_tags(tag_id)`
カラム定義:
|列|型|NULL|説明|
|---|---|---|---|
|issue_id|INTEGER|NO|課題ID|
|tag_id|INTEGER|NO|タグID|
|created_at|TEXT|NO|作成日|

**主要リレーション**
- `teams` 1 - N `projects`
- `teams` 1 - N `team_members` N - 1 `users`
- `projects` 1 - N `milestones`
- `projects` 1 - N `issues`
- `issues` 1 - N `issues`（親課題と子課題）
- `issues` N - N `tags`（`issue_tags` を介す）

**進捗算出の前提**
- 進捗は「完了した予定時間 ÷ 予定時間合計」で算出する。
- 実績時間は記録するが、進捗算出には使用しない。

**インデックス方針（補足）**
- ガント表示や一覧表示のため、`project_id + planned_start/planned_end` を起点に取得できるようにする。
- 未着手一覧（バックログ）は `planned_start IS NULL AND planned_end IS NULL` を前提にし、`project_id + priority` で並べ替えられるようにする。
- 担当者別の抽出に `assignee_user_id` を使う。

**要確認事項**
- WBS 階層は「課題（チケット）→ 子課題（タスク）」の2段固定でよいか。
- 状態（`status`）の取り得る値を `未着手/進行中/完了` で固定してよいか。
- 優先度のスケール（例: 1〜5）と「小さいほど高い」ルールでよいか。
- タグのスコープはチーム単位でよいか、プロジェクト単位が必要か。
- デイリータスクは「日次で未完了に戻す」タイミングをどう定義するか。
- 週次実績の集計起点は `completed_at` でよいか、別の完了日定義が必要か。
