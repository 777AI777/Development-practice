# 開発フロー（並行開発前提）

## 前提
- 対象リポジトリ: Dev_honda/task-app
- 用語は Backlog 準拠（マイルストーン/課題/子課題）
- 進捗は予定ベースで算出する
- 複数ユーザーでチーム管理を行う
- ガント/一覧/未着手一覧（バックログ）を提供する

## 開発フェーズ（順序）
1. 要件確認・用語整備
目的: 要件定義の内容と用語を確定し、Backlog用語で統一する
成果物: 用語集、非機能制約の確定、要確認事項の一覧
並行可: なし
依存: なし
2. 情報設計（画面・データ・権限）
目的: 画面遷移、WBS階層、権限モデル、データ構造を設計する
成果物: 画面概要、ER/データ項目、権限方針
並行可: 画面概要とデータ設計の作成
依存: フェーズ1完了
3. WBS・スケジュール設計
目的: マイルストーン/課題/子課題の運用ルール、進捗算出、未着手一覧（バックログ）運用を定義する
成果物: WBS運用ルール、進捗算出仕様、バックログ運用仕様
並行可: ガント表示仕様と一覧ビュー仕様の整備
依存: フェーズ2完了
4. 基盤実装（認証・チーム・DB）
目的: 複数ユーザー/チーム管理の基盤とSQLite永続化を構築する
成果物: 認証、チーム/権限、DBスキーマ
並行可: 認証実装とDBスキーマ実装
依存: フェーズ2完了
5. 機能実装（WBS・進捗・表示）
目的: WBS管理、進捗管理、ガント/一覧/未着手一覧を実装する
成果物: WBS操作、進捗算出、各ビュー
並行可: WBS操作、進捗算出、ガント/一覧/未着手一覧の各実装
依存: フェーズ3完了、フェーズ4完了
6. 仕上げ実装（付加機能）
目的: タグ/色、デイリータスク、完了演出、週次実績を実装する
成果物: 付加機能一式
並行可: タグ/色、デイリータスク、週次実績の各実装
依存: フェーズ5完了
7. 結合確認・受け入れ準備
目的: 予定ベース進捗の整合性、主要画面の結合を確認する
成果物: 結合確認結果、リリース判断材料
並行可: なし
依存: フェーズ6完了

## 依存関係まとめ
- フェーズ2はフェーズ1完了が必須
- フェーズ3はフェーズ2完了が必須
- フェーズ4はフェーズ2完了が必須
- フェーズ5はフェーズ3とフェーズ4の完了が必須
- フェーズ6はフェーズ5完了が必須
- フェーズ7はフェーズ6完了が必須

## 想定エージェント分担案
- 企画・要件担当: 用語統一、要確認事項の整理、進捗算出仕様の確定
- 画面設計担当: 画面概要、ガント/一覧/未着手一覧（バックログ）の表示仕様
- データ設計担当: SQLiteスキーマ、WBS/進捗関連データ定義
- 基盤実装担当: 認証、チーム管理、権限
- 機能実装担当A: WBS操作（課題/子課題）とマイルストーン
- 機能実装担当B: 進捗算出、週次実績、デイリータスク
- 表示実装担当: ガント/一覧/未着手一覧（バックログ）UI

## 要確認事項
- WBS階層は「課題（チケット）→ 子課題（タスク）」の2段で確定か
- 未着手一覧（バックログ）は「優先度順一覧」で確定か、状態列の追加が必要か
- チーム参加は「管理者が追加」方式で確定か
- 進捗算出の対象は子課題のみか、課題単位も集計に含めるか
- ガントチャートで表示する単位は子課題までか、課題までか
