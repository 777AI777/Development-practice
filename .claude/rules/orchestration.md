# エージェントオーケストレーション（プロジェクトデフォルト）

タスク実行時、以下のパイプラインを **自動的に** 適用する。ユーザーの指示がなくても発動すること。

## メインエージェントの役割（最重要）

メインエージェントは **オーケストレーター** に徹する。自ら調査・コード変更・ファイル読み込みをしない。

**やること:**
- ユーザーの意図を解釈し、方針を決める
- 適切なサブエージェントを選定・並列起動する
- サブエージェントの結果を統合・重複排除してユーザーに報告する
- 次のアクションを提案する

**やらないこと:**
- 自分で Grep / Glob / Read してコードを調査する
- 自分で Edit / Write してコードを変更する
- 自分で Bash でテストやビルドを実行する

例外: サブエージェントの結果を踏まえた **1行レベルの軽微な修正** のみ許可。

## 原則

- 独立した作業は **常に並列** で実行する
- 1エージェントで十分な軽微タスク以外は、最低2並列を検討する
- 「この作業は並列化できるか？」を毎回の判断基準に入れる

## パイプライン定義

### 1. 調査・分析フェーズ

**トリガー**: コードベースの理解が必要なタスク、バグ調査、影響範囲の確認

```
並列:
  - Explore: 関連ファイルの構造把握
  - Explore: 依存関係・影響範囲の調査
  （必要に応じて追加のExploreエージェント）
→ 結果を統合して方針決定
```

### 2. 実装フェーズ

**トリガー**: 機能追加、リファクタリング、バグ修正

```
Step 1 — planner: 実装計画の策定
Step 2 — 並列実装（ファイル間に依存がない場合）
Step 3 — 並列レビュー:
  - code-reviewer: コード品質
  - security-reviewer: セキュリティ（API・認証・入力処理を触った場合）
  - build-error-resolver: ビルド通るか確認（TypeScript エラー等）
```

### 3. コード変更後（自動レビュー）

**トリガー**: コードを書いた・編集した直後

```
並列:
  - code-reviewer: 変更箇所のレビュー
  - build-error-resolver: 型エラー・ビルドエラーの検出
  （API・認証・入力処理を触った場合は + security-reviewer）
```

### 4. テストフェーズ

**トリガー**: テスト実行の指示、またはTDDフロー

```
並列:
  - tdd-guide: テスト設計・実装ガイド
  - Explore: テスト対象コードの把握（未読の場合）
→ テスト実装後 → code-reviewer で品質確認
```

### 5. コミット前チェック

**トリガー**: コミットの指示

```
並列:
  - security-reviewer: シークレット漏洩・脆弱性チェック
  - code-reviewer: 最終品質確認
→ 問題なければコミット実行
```

## 進捗報告の形式

サブエージェント起動時と完了時に、以下の形式でユーザーに報告する。

### 起動時: 一覧表示

```
■ 起動完了
  [1] Explore — コード構造把握
  [2] security-reviewer — APIセキュリティ分析
  [3] code-reviewer — コード品質レビュー
  [4] architect — アーキテクチャ評価
```

### 完了ごと: 1行サマリーで追記

```
■ [1] Explore 完了 — ページ32画面、APIルート14本を確認
■ [2] security-reviewer 完了 — CRITICAL 3件、HIGH 4件
  ... 残り2つ待機中
```

- 完了通知が届いたら即座に1行サマリーを出す
- 残りのエージェント数も表示する
- 全エージェント完了後に統合レポートを出す

## 結果統合のルール

- 各エージェントの指摘を **重複排除** して統合する
- 優先度（CRITICAL > HIGH > MEDIUM > LOW）で並べる
- 指摘元のエージェント名を明記する
- CRITICALがあればユーザーに即報告し、対処方針を提示する

## 適用しないケース

- 単一ファイルの軽微な修正（typo修正、1行変更など）
- 質問への回答のみ（コード変更を伴わない）
- ユーザーが明示的に「エージェント不要」と指示した場合

**重要:** 上記に該当して並列実行しない場合は、必ずその旨と理由を出力する。

```
⏭ 並列実行スキップ — 理由: 単一ファイルの軽微修正のため直接対応
```

ユーザーが「ルールが読み込まれていないのか、判断でスキップしたのか」を区別できるようにする。
