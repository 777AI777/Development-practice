## 3. Phase 2 新規・拡張機能（公開4〜8ヶ月目・目標500〜3,000人）

### 3.1 外部DB連携サジェスト

Phase 1ではフリーテキスト入力のみで運用し、データ蓄積を優先。Phase 2で導入する。

**要件:**
- リスト要素の入力時、テキスト入力に応じて外部DBから候補をサジェスト表示
- 連携先DB（優先度順）:
  - 音楽: Spotify API（アーティスト名・曲名・アルバム名）
  - 映画/ドラマ: TMDB API（作品名）
  - 本: Google Books API（書籍名）
  - 飲食店/場所: Google Places API（店名・スポット名）
- サジェストから選択した場合、正規化された名称＋メタデータ（ジャンル等）が構造化データとして保存される
- サジェストに該当がない場合はフリーテキストとして保存可能（抽象的なテーマ「雨の日の過ごし方」等に対応）
- サジェストの利用は任意であり、強制しない

**理由:** サジェストにより入力の摩擦を下げ、表記ゆれを防ぎ、データの構造化品質を向上させる。レコメンド・検索の精度に直結する。

### 3.2 「〜が好きな人が好きなもの」検索

ユーザー数・投稿データが十分に蓄積されないと検索精度が機能しないため、Phase 2で導入する。

**要件:**

検索:
- 検索ボックスにキーワードを入力（例:「PEOPLE1」「インターステラー」「スピッツ」）
- 入力時にサジェスト表示（3.1のDB連携と同じサジェスト基盤を利用）
- 検索結果画面に「〇〇を挙げている人が他に挙げているもの」を表示
- 表示形式: 頻度順のリスト（「○○を挙げた人の○%が△△も挙げています」）
- ジャンルをまたいだ結果も表示（音楽を検索→カフェや映画の結果も出る）

閲覧専用での利用:
- 検索・閲覧はアカウント登録なしでも可能
- 未登録・未投稿ユーザーでも検索結果を見られる
- 検索行動自体はログとして記録（需要の可視化、将来のレコメンド改善に活用）

データ要件:
- 一定数以上の投稿データが蓄積されるまでは検索精度が低いため、初期は「データ蓄積中」等の表示で対応
- Phase 1のコールドスタート対策と連動し、ユーザー数増加に応じて閾値を引き上げ

### 3.3 ネタバレ配慮トグル

**要件:**
- 投稿画面に「ネタバレを含む」のON/OFFトグルを1つ設置（デフォルトOFF）
- ONの場合、タイムラインおよび検索結果ではテーマ（タイトル）のみ表示し、リスト内容はぼかし処理
- ユーザーがタップすると内容が表示される
- プロフィール画面での自分の投稿一覧では常に表示（自分の投稿は本人にはぼかさない）

### 3.4 自己分析ツール — ユーザー間比較

自己分析ツールの基本6メニュー（タグTOP3・カテゴリ分布・ヒートマップ・タグ変遷・好みDNA・年間振り返り）は**すべてPhase 1で実装済み**。Phase 2では、Phase 1の自己分析基盤を前提とした「ユーザー間比較」機能のみを追加する。

**要件:**
- 他ユーザーと3件一致の検出
- 好みが近い人の発見
- 変動報酬タイミング設計は不採用

> **補足:** Phase 2以降で自己分析メニューの追加があり得るが、現時点では未定（要検討）。

### 3.5 スポンサード投稿

**要件:**
- フィードネイティブ広告として実装
- 公式アカウント（Phase 3の3.8参照）との連携を前提
- 詳細はPhase 2開始時に設計

### 3.6 お気に入り日記（プッシュ通知型お題）

Phase 1対象外。Phase 2以降で検討する。

**概要:** ユーザーが自分で設定した頻度で、お題形式の通知が届き記録を促す機能。

**要件:**

通知設定:
- デフォルトはOFF
- ユーザーが設定画面から有効化
- 頻度選択:「毎日」「週1回」「通知なし」
- 通知時刻の指定（デフォルト: 21:00等）

お題の内容:
- 運営があらかじめ用意したお題プールからランダムに出題
- お題例:「今日のお気に入りは?」「最近ハマっているものは?」「今日食べて美味しかったものは?」
- おすすめテーマとは別管理（日記は個人向け、おすすめテーマは全体向け）

回答:
- 通知からアプリを開き、通常の投稿フォーマット（テーマ＋3つ）で回答
- 回答はプロフィールに蓄積される（通常投稿と同列）
- 日記経由の投稿であることを示すタグ/ラベルを付与（任意表示）

## 4. Phase 3 機能（公開9〜12ヶ月目・目標3,000〜10,000人）

### 4.1 公式アカウント機能

ユーザー基盤が3,000人以上になってから導入。Phase 1〜2では通常アカウントで運営が代行する形で先行テスト可能。

**要件:**

公式アカウント:
- 通常ユーザーアカウントとは別に「公式アカウント」種別を設ける
- 公式アカウントには認証バッジ（公式マーク）を表示
- 公式アカウントの発行は運営による手動承認制

公式スレッド:
- 公式アカウントが作成したスレッドは通常スレッドと同じフォーマット
- 公式スレッドのトップ掲出機能: 運営設定により、特定の公式スレッドをスレッド一覧・ホーム画面の上部に固定表示可能（広告枠として提供）
- 公式アカウントのプロフィールページに、そのアカウントが作成した全スレッドを一覧表示

### 4.2 横断検索の精度改善

Phase 2で導入した「〜が好きな人が好きなもの」検索の精度を、データ量増加に合わせて改善。

### 4.3 データの外部提供（課金制）

蓄積された好みデータの外部提供。詳細はPhase 3開始時に設計。

## 10. 最終到達地点（Phase 3の先）

Phase 3で10,000人に到達した後も、OKINYの目標はそこで終わらない。

**ビジョン:** 一人ひとりが自分の「好き」を理解し、広げられる世界を作る。

**好みの世界地図:** ジャンルを超えた「好き」の繋がりが可視化された、世界に類のないデータベースを構築する。「あるバンドが好きな人は、どんな映画を観て、どんなカフェに行くのか」——この問いに答えられるプラットフォームは、まだどこにもない。

**数億円規模の資産価値を持つデータ基盤へ:** ユーザーが「好き」を投稿するたびにデータベースは成長し続ける。蓄積された好みのデータは、メディア・EC・エンタメ業界にとって唯一無二の資産になる。後から同じものを作ろうとしても、この蓄積は追い越せない。
