# OKINY 追加要件定義書

---

## 1. 本書の目的

今回の構想議論を踏まえ、既存のOKINYアプリに対して追加・変更すべき機能要件を定義する。
既存機能（フォロー、プロフィール蓄積、投稿機能、外部シェア）は原則維持し、差分のみを記載する。

---

## 2. 既存機能の変更

### 2.1 投稿フォーマットの変更：ランキング形式 → 並列リスト形式

**現状：** 投稿がランキング形式（1位/2位/3位の順位あり）
**変更後：** 順位を排除し、「テーマ＋好きなもの3つ」の並列リスト形式に変更

**要件：**
- 投稿時に順位番号（1st/2nd/3rd等）を表示しない
- 3つの要素は並列に表示される（上下の並び順は入力順だが、順位を示すUIは付けない）
- テーマ（タイトル）の入力欄を必須項目として追加（例：「雨の日に聴きたい曲」）
- テーマは自由入力（文字数上限は要検討、目安30文字程度）
- リスト要素は3つ固定（2つ以下・4つ以上は不可）

**理由：** 順位をつけることは「優劣の強制」「共有のハードル」を生む。並列にすることで、その瞬間の純粋な熱量を切り取るタイムカプセルとして機能させる。

---

### 2.2 投稿の外部シェアUI

**現状：** シェア機能はあるがUIは未定
**変更後：** 統一フォーマットの投稿カード画像を自動生成し、ワンタップでシェア

**要件：**
- 投稿完了時またはプロフィール画面から「シェア」ボタンでカード画像を生成
- カード画像のレイアウトは統一（テーマ＋3つのリスト＋OKINYロゴ）
- 骨格（レイアウト・ロゴ配置）は固定、装飾（背景色・アクセントカラー）はユーザーが選択可能
- シェア先：X、Instagram Stories、LINEを想定（OS標準のシェアシート経由）
- カードを見た人がOKINYに興味を持てるよう、アプリストアへのリンクまたはQRを含める

**理由：** 投稿カードが外部SNSで「OKINYだ」と一目でわかるブランド認知の核になる。Topsters画像がXで流行したのと同じバイラル導線を設計する。

---

### 2.3 投稿時の外部DB連携サジェスト【Phase 2】

**現状：** リスト要素は自由テキスト入力のみ（想定）
**変更後：** 外部DBからのサジェスト＋フリーテキストの混合型
**実装時期：** Phase 2。Phase 1ではフリーテキスト入力のみで運用し、データ蓄積を優先する。

**要件：**
- リスト要素の入力時、テキスト入力に応じて外部DBから候補をサジェスト表示
- 連携先DB（優先度順）：
  - 音楽：Spotify API（アーティスト名・曲名・アルバム名）
  - 映画/ドラマ：TMDB API（作品名）
  - 本：Google Books API（書籍名）
  - 飲食店/場所：Google Places API（店名・スポット名）
- サジェストから選択した場合、正規化された名称＋メタデータ（ジャンル等）が構造化データとして保存される
- サジェストに該当がない場合はフリーテキストとして保存可能（抽象的なテーマ「雨の日の過ごし方」等に対応）
- サジェストの利用は任意であり、強制しない

**理由：** サジェストにより入力の摩擦を下げ、表記ゆれを防ぎ、データの構造化品質を向上させる。レコメンド・検索の精度に直結する。

---

### 2.4 ネタバレ配慮トグル

**現状：** なし
**変更後：** 投稿時にネタバレトグルを設置

**要件：**
- 投稿画面に「ネタバレを含む」のON/OFFトグルを1つ設置（デフォルトOFF）
- ONの場合、タイムラインおよび検索結果ではテーマ（タイトル）のみ表示し、リスト内容はぼかし処理
- ユーザーがタップすると内容が表示される
- プロフィール画面での自分の投稿一覧では常に表示（自分の投稿は本人にはぼかさない）

## 3. 新規機能

### 3.1 スレッド機能（知恵袋型Q&A）

**概要：** ユーザーがお題（質問）を立て、他のユーザーがリスト形式で回答するQ&A機能。

**要件：**

スレッド作成：
- 任意のユーザーがスレッドを作成可能
- スレッドのタイトル（お題）を入力（例：「PEOPLE1が好きな人におすすめのバンド教えて」）
- 補足テキスト（任意、200文字程度上限）を入力可能
- 作成したスレッドはタイムラインおよびスレッド一覧に表示

回答：
- 回答は通常投稿と同じフォーマット（好きなもの3つの並列リスト）
- 回答にはテーマ不要（スレッドのお題がテーマを兼ねる）
- 1ユーザー1スレッドにつき1回答（重複投稿不可）
- 回答はスレッド内に蓄積されるとともに、回答者のプロフィールにも同じ形式で蓄積される

コミュニケーション設計（防音壁モデル）：
- スレッド主と各回答者の間でのみ1対1のコメントが可能
- 回答者同士のコメント・やり取りは不可
- 外野（未回答ユーザー）からのコメントは不可（閲覧は可能）
- コメントはテキストのみ（画像・スタンプ等は不要）
- コメントの文字数上限：200文字程度

スレッド一覧・検索：
- スレッド一覧画面を新設（タブまたはメニュー項目）
- キーワード検索、新着順・回答数順の並び替え
- 回答数の表示

---

### 3.2 今日のお題（運営発の全体質問）

**概要：** 運営が全ユーザーに共通のお題を投げ、全員が同じ問いに対して回答する機能。

**要件：**

お題の配信：
- 運営（管理者権限）のみがお題を作成・配信可能
- 配信タイミングは手動（固定スケジュールではない）
- お題のフォーマット：タイトル（例：「今一番ハマっている食べ物は？」）＋補足テキスト（任意）
- 配信時にプッシュ通知を送信（ユーザーがプッシュ通知を許可している場合）

回答：
- 回答フォーマットはスレッドと同じ（好きなもの3つの並列リスト）
- 1ユーザー1お題につき1回答
- 回答は「今日のお題」画面内に蓄積されるとともに、回答者のプロフィールにも蓄積

閲覧：
- 全員の回答が一覧で閲覧可能（新着順）
- 回答後に即時フィードバック表示（3.5のマイルストーンとは別に）：
  - 「あなたと同じものを挙げた人は○人中○人でした」
  - 「この組み合わせはあなただけです」等
- 過去のお題はアーカイブとして閲覧可能

コミュニケーション：
- 今日のお題への回答に対するコメント機能はなし（閲覧のみ）
- 理由：全員参加型のため、コメントを許可すると荒れるリスクがある

UI上の配置：
- ホーム画面上部またはタブとして目立つ位置に配置
- 未回答のお題がある場合はバッジ等で通知

---

### 3.3 チーム（グループ）機能

**概要：** 仲間内でグループを作り、メンバーの好きの重なり・違いを可視化する機能。

**要件：**

チーム作成：
- 任意のユーザーがチームを作成可能
- チーム名を設定（例：「大学の映画サークル」）
- 招待リンクを生成し、共有（LINEやX等で送信）
- 招待リンク経由で参加（アプリ内でチーム参加の承認フロー）
- 1チームのメンバー上限：20人程度（要検討）
- 1ユーザーが参加可能なチーム数の上限：10程度（要検討）

マインドマップ（好みの可視化）：
- チームメンバー全員の投稿データを集約し、好みの重なり・違いを可視化
- 可視化内容（最低限）：
  - 全員が挙げているもの（共通の好き）
  - 自分だけが挙げているもの（ユニークな好き）
  - メンバー間の好みの近さ（「AさんとBさんは○個重なっています」）
- 可視化のビジュアル形式：マインドマップ形式（中心にチーム名、枝分かれして各メンバーの好きが広がる）
  ※具体的なビジュアルデザインはUIデザインフェーズで決定
- マインドマップはチームメンバーのみ閲覧可能
- マインドマップの結果を画像として外部シェア可能（バイラル導線）

---

### 3.4 「〜が好きな人が好きなもの」検索

**概要：** 投稿データを横断検索し、特定のものを好きな人が他に何を好きかを閲覧できる機能。

**要件：**

検索：
- 検索ボックスにキーワードを入力（例：「PEOPLE1」「インターステラー」「スピッツ」）
- 入力時にサジェスト表示（2.3のDB連携と同じサジェスト基盤を利用）
- 検索結果画面に「〇〇を挙げている人が他に挙げているもの」を表示
- 表示形式：頻度順のリスト（「○○を挙げた人の○%が△△も挙げています」）
- ジャンルをまたいだ結果も表示（音楽を検索→カフェや映画の結果も出る）

閲覧専用での利用：
- 検索・閲覧にアカウント登録は必要だが、投稿は不要
- 未投稿ユーザーでも検索結果を見られる
- 検索行動自体はログとして記録（需要の可視化、将来のレコメンド改善に活用）

データ要件：
- 一定数以上の投稿データが蓄積されるまでは検索精度が低いため、初期は「データ蓄積中」等の表示で対応
- 最低表示条件：同じものを挙げているユーザーが3人以上の場合に結果表示（要検討）

---

### 3.5 成功報酬型フィードバック（マイルストーン）

**概要：** 固定の時間軸ではなく、蓄積量や条件達成をトリガーにフィードバックを返す仕組み。

**要件：**

マイルストーンの種類：

| トリガー | フィードバック内容 | 備考 |
|---------|---------------|------|
| 投稿3件達成 | 好みの傾向表示（「音楽系が多いですね」等） | ジャンル比率の簡易分析 |
| 投稿10件達成 | 好みのマインドマップ生成 | 自分の好きの全体像を可視化 |
| 投稿20件達成 | 好みが近いユーザーの発見（「あなたと3つ同じ人がいます」） | 匿名で表示、相手のプロフィールへリンク |
| 投稿50件達成 | 好みレポート完成（Spotifyラップアップ的な体験） | 蓄積の振り返りストーリー |
| 同一ジャンル3件 | ジャンル集中の気づき（「映画が3つ揃いました」） | ジャンルタグの自動付与が前提 |
| 他ユーザーと3件一致 | 好みの共鳴通知（「あなたと3つ同じものを挙げた人がいます」） | 匿名、相手プロフィールへリンク |
| ユニークな組み合わせ | 「この組み合わせはあなただけです」 | 全ユーザー中の一意性判定 |

通知・表示：
- マイルストーン達成時にアプリ内通知（バッジ＋ポップアップ）
- プッシュ通知も送信（ユーザーが許可している場合）
- フィードバック結果を画像として外部シェア可能
- マイルストーン一覧画面を設置（達成済み/未達成の進捗が見える）

設計方針：
- 報酬タイミングは予測不能（変動報酬）であること
- マイルストーンの追加・調整は運営側で柔軟に変更可能な設計とする

---

### 3.6 お気に入り日記（プッシュ通知型お題）

**概要：** ユーザーが自分で設定した頻度で、お題形式の通知が届き記録を促す機能。

**要件：**

通知設定：
- デフォルトはOFF
- ユーザーが設定画面から有効化
- 頻度選択：「毎日」「週1回」「通知なし」
- 通知時刻の指定（デフォルト：21:00等）

お題の内容：
- 運営があらかじめ用意したお題プールからランダムに出題
- お題例：「今日のお気に入りは？」「最近ハマっているものは？」「今日食べて美味しかったものは？」
- お題は「今日のお題」（3.2）とは別管理（日記は個人向け、今日のお題は全体向け）

回答：
- 通知からアプリを開き、通常の投稿フォーマット（テーマ＋3つ）で回答
- 回答はプロフィールに蓄積される（通常投稿と同列）
- 日記経由の投稿であることを示すタグ/ラベルを付与（任意表示）

---

### 3.7 投稿時レコメンド

**概要：** 投稿完了時に、同じものを挙げている人の他の好きをレコメンドとして表示する機能。

**要件：**
- 投稿完了画面に「〇〇を挙げた人はこんなものも好きです」を表示
- 表示件数：3〜5件程度
- 表示条件：同じアイテムを挙げているユーザーが一定数以上いる場合のみ
- 初期（データ蓄積不足時）は表示しない or 「データ蓄積中」と案内
- レコメンドされたアイテムをタップすると、そのアイテムの検索結果画面へ遷移

**理由：** 「投稿したら発見が返ってくる」体験がOKINYの継続動機の核。投稿→発見→次の投稿の「好きの連鎖ループ」を回す。

---

### 3.8 公式アカウント機能

**概要：** 企業・ラジオ番組等が公式アカウントを持ち、スレッドを作成・運営する機能。

**要件：**

公式アカウント：
- 通常ユーザーアカウントとは別に「公式アカウント」種別を設ける
- 公式アカウントには認証バッジ（公式マーク）を表示
- 公式アカウントの発行は運営による手動承認制

公式スレッド：
- 公式アカウントが作成したスレッドは通常スレッドと同じフォーマット
- 公式スレッドのトップ掲出機能：運営設定により、特定の公式スレッドをスレッド一覧・ホーム画面の上部に固定表示可能（広告枠として提供）
- 公式アカウントのプロフィールページに、そのアカウントが作成した全スレッドを一覧表示

フェーズ：
- Phase 2以降（ユーザー基盤が1,000人以上になってから導入）
- Phase 1では通常アカウントで運営が代行する形で先行テスト可能

## 4. 管理者機能（運営側）

### 4.1 今日のお題の管理
- お題の作成・編集・配信・削除
- 配信予約（日時指定での自動配信）
- 過去のお題一覧と回答数の確認

### 4.2 通報・モデレーション
- ユーザーからの通報を受け付け（通報ボタンを各投稿・スレッド・コメントに設置）
- 通報一覧画面（未対応/対応済みのフィルター）
- 投稿・スレッド・コメントの非公開化/削除
- 悪質ユーザーのアカウント停止
- NGワードフィルター（投稿・スレッド作成時に自動検知、投稿をブロック）
- 対応速度のKPI管理（通報から対応までの時間を記録）

### 4.3 公式アカウント管理
- 公式アカウントの発行・停止
- 公式スレッドのトップ掲出設定（掲出期間・優先度の管理）

### 4.4 マイルストーン管理
- マイルストーンの追加・編集・無効化
- トリガー条件と表示テキストの設定変更
- 達成状況の全体統計（何%のユーザーがどのマイルストーンに到達しているか）

### 4.5 お気に入り日記のお題管理
- お題プールの追加・編集・削除
- お題の出題頻度調整

---

## 5. 非機能要件

### 5.1 データ構造
- 投稿データは「テーマ」「リスト要素（3つ）」「各要素のメタデータ（外部DB由来の場合）」「投稿日時」「投稿種別（自由投稿/スレッド回答/今日のお題回答/日記）」を構造化して保存
- 横断検索・レコメンドのためのインデックス設計が必要
- ユーザー間の好み類似度を算出するための集計テーブルまたはキャッシュ機構

### 5.2 プライバシー
- ユーザーは匿名（ニックネーム＋アバター）
- 好み類似度の算出・表示は匿名のまま（実名や個人情報は表示しない）
- チーム機能内でのメンバー表示はニックネーム
- 検索行動のログは運営分析用に保持するが、個別ユーザーに紐づけた外部提供は行わない

### 5.3 パフォーマンス
- 横断検索のレスポンス：3秒以内（目標）
- 投稿時レコメンドの生成：2秒以内（目標）
- マイルストーンの判定はリアルタイムではなくバッチ処理でも可（投稿直後〜数分以内に発火すれば許容）

---

## 6. 優先度と実装フェーズ

### Phase 1（MVP・初期リリース）
最初の200〜1,000人を獲得し、コアな体験を検証するために必要な最小限。

| 機能 | 優先度 |
|------|--------|
| 2.1 投稿フォーマット変更（ランキング→並列） | 必須 |
| 2.2 投稿カードのシェアUI | 必須 |
| 2.4 ネタバレトグル | 必須 |
| 3.1 スレッド機能（防音壁モデル） | 必須 |
| 3.2 今日のお題 | 必須 |
| 3.3 チーム機能（基本版：作成・参加・共通の好き表示） | 必須 |
| 4.2 通報・モデレーション（基本版） | 必須 |
| 4.1 今日のお題の管理 | 必須 |

※Phase 1ではリスト要素の入力はフリーテキストのみ。外部DB連携サジェストはPhase 2で導入。

### Phase 2（中期・機能拡充）
データ蓄積が進み、レコメンドや検索が機能し始めるフェーズ。

| 機能 | 優先度 |
|------|--------|
| 2.3 外部DB連携サジェスト | 高 |
| 3.4 「〜が好きな人が好きなもの」検索 | 高 |
| 3.5 成功報酬型フィードバック（マイルストーン） | 高 |
| 3.7 投稿時レコメンド | 高 |
| 3.3 チーム機能（マインドマップの可視化拡張） | 中 |
| 3.6 お気に入り日記 | 中 |
| 4.4 マイルストーン管理 | 中 |

### Phase 3（長期・拡大）
ユーザー基盤が安定し、マネタイズや外部連携を本格化するフェーズ。

| 機能 | 優先度 |
|------|--------|
| 3.8 公式アカウント機能 | 高 |
| 4.3 公式アカウント管理 | 高 |
| 横断検索の精度改善（AI/ML） | 中 |
| API外部提供（課金制） | 中 |
| プレミアム機能（有料プラン） | 中 |

## 7. 備考

- 本書はアプリの「追加・変更」要件のみを記載しており、既存機能（アカウント管理、基本的なタイムライン、フォロー機能等）の仕様は対象外とする
- 各機能の詳細UI/UX設計は別途UIデザインフェーズで定義する
- 外部API連携の利用料金・利用制限は別途技術調査が必要
- マイルストーンの具体的な閾値（3件/10件/20件/50件等）は運用開始後にユーザーの投稿頻度を見て調整する
