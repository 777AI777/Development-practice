# OKINYプロフィール蓄積可視化 検討

## 背景・課題

OKINYは「テーマ＋好きなもの3つ」を並列リスト（順位なし）で投稿するSNS。  
投稿例：テーマ「雨の日に聴きたい曲」→ スピッツ - 楓 / Mr.Children - Rain / BUMP OF CHICKEN - 天体観測

投稿はプロフィールに蓄積されるが、投稿一覧がただ並ぶだけだと「溜まっていく実感」が薄い。蓄積のモチベーションになるような可視化を、投稿1件目から機能する形で常設したい。

### 制約

- Phase 1時点ではリスト要素はフリーテキスト入力（外部DB連携なし）
- ジャンルの自動判別はできない（ユーザーがタグを選ぶのみ）
- 凝りすぎると開発コストが膨らむ。Phase 1で最小限に実装できる範囲

### 参考事例

| サービス | 可視化手法 | 効果 |
|---------|-----------|------|
| Letterboxd | 年間鑑賞本数＋ポスターグリッド | 蓄積体験の可視化で1,000万ユーザーを獲得 |
| Spotifyラップアップ | 年1回の振り返りサマリー | SNSシェアで大きなバイラルを生成 |
| 読書メーター | 冊数・ページ数・著者円グラフ | 読書量の定量的可視化 |

## アイデア5つ

### アイデア2：タグ別投稿数バーチャート

**概要**  
ユーザーがよく使うタグの投稿数を横棒グラフで可視化する。「自分はどのジャンルが好きか」を一目で確認できる。

**ユーザー体験**  
「映画：23件」「音楽：15件」「カフェ：8件」のようなバーチャートを見て、自分の興味分布を客観視できる。「食べ物タグが意外と少ない」など、新しい自己発見につながる。

| 項目 | 評価 |
|------|------|
| モチベーション効果 | 中 |
| 開発コスト | 小 |
| Phase1実装可否 | 可 |

**実装メモ**  
- `rankings` テーブルの `tag_id` でGROUP BYしてカウント。既存の `GET /api/v1/rankings` にタグ情報は含まれているため、クライアント側で集計可能
- グラフはCSSのみで実装可能（`width: {count/max*100}%` のインラインスタイル）。外部チャートライブラリ不要
- 上位5〜7タグのみ表示し、残りは「その他」にまとめる
- タグが0件の場合は非表示にすることでコールドスタート問題を回避

### アイデア4：「好きなもの」頻出ワードランキング

**概要**  
投稿内のリスト要素（フリーテキスト）を形態素解析なしに単純な文字列マッチで集計し、よく登場するワードをランキング形式で表示する。「あなたの好きなもの TOP5」。

**ユーザー体験**  
「スピッツ：5回」「ラーメン：3回」のようなリストを見て、「自分はこれが好きなんだな」と再確認できる。フォロワーが見たときに「この人はスピッツが好きなんだ」と人となりが伝わる。

| 項目 | 評価 |
|------|------|
| モチベーション効果 | 中 |
| 開発コスト | 中 |
| Phase1実装可否 | 条件付き可 |

**実装メモ**  
- `ranking_items.item_text` を全取得し、フロントで文字列集計
- 前処理：全角・半角の統一、句読点・記号除去、空白トリム程度でよい（形態素解析は不要）
- 完全一致または部分一致でカウント。「スピッツ」と「スピッツ - 楓」は別カウントになる課題あり
- 投稿件数が少ないうちは「まだデータが少ないです（5件以上で表示）」のような閾値を設ける
- **注意点**：フリーテキストのため表記ゆれが多く、精度は低め。過大な期待をさせない見せ方が必要

## フリーテキスト制約下での可視化の可能性

### できること

| 可視化内容 | 理由 |
|-----------|------|
| 投稿数カウント・ストリーク | メタデータ（`created_at`）で完結 |
| タグ別分布チャート | `tag_id` の集計のみ |
| カレンダーヒートマップ | `created_at` の日付集計のみ |
| 月別・週別サマリー | 同上 |
| 完全一致での頻出ワード集計 | フロントでの文字列マッチで対応可 |

### 工夫次第でできること

| 可視化内容 | 必要な工夫 |
|-----------|-----------|
| 表記ゆれを吸収した頻出ワード | 正規化ロジック（全角半角統一、記号除去）をフロントに実装 |
| 「好きなアーティスト」推定 | 「アーティスト名 - 曲名」形式のパターンマッチ |
| 感情・テンション分析 | テーマテキストのキーワード辞書マッチ（「癒し」「エモい」等） |

### Phase2以降でないとできないこと

| 可視化内容 | 理由 |
|-----------|------|
| 正確なジャンル分類 | 外部DB（MusicBrainz、TMDb等）との連携が必要 |
| 楽曲・映画のサムネイル表示 | 著作権・API利用規約の問題 |
| 他ユーザーとの比較 | フォロー・ソーシャルグラフが必要 |
| AIによる趣味の要約文生成 | LLM API呼び出し（コスト・レイテンシの問題） |
| 精度の高い自然言語処理 | 形態素解析エンジン（MeCab等）の導入が必要 |

## 「溜まっていく感」の最低要件

### 必須要素（これがないと蓄積感が出ない）

- **数字の変化が目に見える**：投稿数が増えるたびに数値が変わること。「83→84件」の変化でもモチベーションになる
- **時間軸の表現**：「いつ投稿したか」が見えること。カレンダーや月別サマリーがこれに該当する
- **投稿1件目から機能すること**：コールドスタートで空白が表示されると離脱につながる

### あると良い要素（体験を豊かにする）

- **自己発見の驚き**：「自分はこんなに映画が好きだったのか」という気づき（タグ分布、頻出ワード）
- **継続への引力**：ストリーク、月別比較など、途切れると惜しいと感じる仕組み
- **シェアしたくなる見た目**：綺麗にまとまったビジュアルで、スクリーンショットをシェアしたくなる

### 過剰要素（Phase1では不要）

- **外部データとの連携**（アルバムアート、映画ポスターなど）：開発コスト大かつPhase1のスコープ外
- **AIによる文章生成**（「あなたの好きなものを一言で表すと…」）：LLMコスト・品質担保が難しい
- **複雑なインタラクション**（ドリルダウン、フィルター切り替えなど）：必要最低限のUIで十分
- **リアルタイム更新**：ページロード時の集計で十分

## 推奨案

### Phase1推奨実装セット

**セット：アイデア1（ストリーク）＋ アイデア5（月別サマリー）＋ アイデア2（タグ別チャート）**

| # | アイデア | 理由 |
|---|---------|------|
| 1 | 投稿数カウンター＋ストリーク | 開発コスト最小で即効性が高い。毎日の習慣化に直結する |
| 2 | 月別投稿サマリーカード | 振り返りとシェア訴求の両立。月次の区切りが自然な達成感を生む |
| 3 | タグ別投稿数バーチャート | 自己発見の要素として重要。外部ライブラリ不要でCSSのみ実装可能 |

### 推奨しない理由

- **アイデア3（カレンダーヒートマップ）**：効果は高いが、`7×52` グリッドのレスポンシブ対応がモバイルで難しく、工数に対してROIが低い。Phase2以降で実装を推奨
- **アイデア4（頻出ワードランキング）**：フリーテキストの表記ゆれ問題が大きく、精度が低いまま見せるとユーザー体験を損なうリスクがある。Phase2でのタグ付き入力との連携後に再検討

### 実装優先順位

```
Week 1: アイデア1（ストリーク）— DBへのアクセス不要、フロント計算のみ
Week 1: アイデア5（月別サマリー）— 既存APIの再利用、新規エンドポイント不要
Week 2: アイデア2（タグ別チャート）— グループ化クエリ追加、CSSチャート実装
```

### 期待効果

- 開発工数：合計3〜5人日（フロントのみ、新規APIエンドポイント最小限）
- ユーザー行動への影響：ストリークによるDAU向上、月別サマリーのシェアによる新規流入
- コールドスタート：全アイデアが投稿1件目から機能する
