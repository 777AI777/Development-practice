# OKINY Phase 1 要件定義書

## 1. 概要

### 1.1 サービス概要

OKINYは、日常の「好き」を整理・共有するSNSアプリ。
投稿フォーマットは「テーマ + 好きなもの3つ（順位なし・並列）」。順位をつけない設計により、優劣の強制を排除し、その瞬間の純粋な熱量を切り取るタイムカプセルとして機能する。

### 1.2 Phase 1の目標

SNSとして成立する最小セットをリリースし、コアな体験を検証する。
個人利用フェーズ（旧Phase 1）は完了済み。Phase 1ではSNS化に必要な機能を追加し、ユーザー間のつながりと継続利用を生む基盤を構築する。

### 1.3 対象ユーザー

- 20〜40代のSNSに馴染みがある多趣味層
- ジャンルを横断して「好き」を整理したいユーザー
- 発信頻度は高くなくても、自分用に記録・管理したいユーザー

### 1.4 UI上の用語

| 内部用語 | UI表示 | 備考 |
|---------|--------|------|
| ranking | 投稿 | UI上は「投稿」で統一。「棚」「ランキング」は使わない |

## 3. Phase 1 新規実装機能

### 3.1 投稿フォーマット変更

**現状**: ランキング形式（1位/2位/3位の順位あり、最大5位）
**変更後**: テーマ + 好きなもの3つの並列リスト形式

**要件**:
- 順位番号（1st/2nd/3rd等）を表示しない
- 3つの要素は並列に表示（並び順は入力順だが、順位を示すUIは付けない）
- テーマ（タイトル）の入力欄を必須項目として追加（例:「雨の日に聴きたい曲」）
- テーマは自由入力（文字数上限は要検討、目安30文字程度）
- リスト要素は3つ固定（2つ以下・4つ以上は不可）
- リスト要素の入力はフリーテキストのみ（外部DB連携サジェストはPhase 2）

**影響範囲**: CRUD全体、下書き、autosave、OGP画像生成、DB（`ranking_items`）

**理由**: 順位をつけることは「優劣の強制」「共有のハードル」を生む。並列にすることで、その瞬間の純粋な熱量を切り取るタイムカプセルとして機能させる。

### 3.2 おすすめテーマ

ユーザーの投稿を促すためのテーマ推奨機能。

**要件**:
- 運営発2〜3本 + ユーザー発3〜4本の混合構成
- カード形式で5〜7枚表示
- 週次でシャッフル（固定表示ではない）
- ラベル・カウント非表示（「○人が回答」等の数値は見せない）
- ホーム画面またはタブとして目立つ位置に配置

**設計方針**: テーマを「お題」として提示し、投稿のきっかけを作る。数値を見せないことで、人数に左右されない自由な参加を促す。

### 3.3 投稿時レコメンド

投稿フォーム内にテーマのレコメンドを表示する機能。

**要件**:
- 投稿フォーム内に「こんなテーマはどう?」的なレコメンドを表示
- 表示件数: 3〜5件程度
- 表示条件: 十分なデータが蓄積されている場合のみ
- 初期（データ蓄積不足時）は表示しない or 「データ蓄積中」と案内
- レコメンドされたテーマをタップすると、テーマ入力欄にセットされる

**理由**: 「投稿したら発見が返ってくる」体験がOKINYの継続動機の核。投稿 → 発見 → 次の投稿の「好きの連鎖ループ」を回す。

### 3.4 スレッド機能

QA型ギャラリー。ユーザーがお題（質問）を立て、他のユーザーがリスト形式で回答する。

**スレッド作成**:
- 任意のユーザーがスレッドを作成可能（作成制約なし）
- スレッドのタイトル（お題）を入力（例:「PEOPLE1が好きな人におすすめのバンド教えて」）
- タイムラインおよびスレッド一覧に表示

**回答**:
- 回答 = 好きなもの3つの並列リスト + ひとこと（任意、140字上限）
- テーマ不要（スレッドのお題がテーマを兼ねる）
- 回答はスレッド内に蓄積されるとともに、回答者のプロフィールにも蓄積

**コミュニケーション設計**:
- 対話不可（投稿主のリアクションのみ）
  - スレッド主から各回答者へのリアクション（スタンプ等）は可能
  - 回答者同士のやり取りは不可
  - 外野（未回答ユーザー）からのリアクションは不可（閲覧は可能）

**ピン止め**:
- スレッド主が回答をピン止め可能
- ピン止めは複数可（上限は要検討）
- ピン止めされた回答はスレッド上部に固定表示

**時間差表示**:
- 回答は即時表示ではなく、一定時間経過後に表示（※詳細は今後詰める）
- 他の回答に引っ張られない、純粋な自分の好きを投稿させるための設計

**Award機能**:
- スレッド主が特に良い回答にAwardを付与可能（※詳細は今後詰める）

**スレッド一覧・検索**:
- スレッド一覧画面を新設
- キーワード検索、新着順・回答数順の並び替え
- 回答数の表示

### 3.5 自己分析ツール（ポイント消費型）

ユーザーの投稿データをもとに、好みの傾向を分析・可視化する機能。ポイントを消費して解放する。

**ポイント獲得**:
| アクション | 獲得ポイント |
|-----------|------------|
| 投稿 | 1pt |
| スレッド回答 | 1pt |
| ピン止めされた | +3pt |
| Award受賞 | +○pt（※詳細は今後詰める） |

**分析メニュー**:
| メニュー | 消費ポイント | 内容 |
|---------|------------|------|
| タグTOP3 | 5pt | よく使うタグの上位3つ |
| カテゴリ分布 | 10pt | ジャンル別の投稿分布 |
| ヒートマップ | 15pt | 投稿頻度の時間帯・曜日分布 |
| タグ変遷 | 25pt | タグの使用傾向の時系列変化 |
| 好みDNA | 40pt | 好みの特徴を総合的に分析（※詳細は今後詰める） |
| 年間振り返り | 50pt | 1年間の投稿を振り返るレポート |

**設計方針**: 投稿・貢献を重ねるほど自己分析が解放される仕組みで、継続利用のインセンティブを作る。

### 3.6 ポイントシステム

自己分析ツールの消費通貨として機能するポイントの管理基盤。

**要件**:
- ポイントの獲得（3.5のアクション）と消費（分析メニュー）を管理
- ポイント残高の表示
- ポイント課金あり（リアルマネーでのポイント購入。レートは未定）
- プレミアムプランではポイント2倍速

### 3.7 プッシュ通知

**要件**:
- Phase 1で実装
- ユーザーの許可制
- 通知対象（※詳細は今後詰める）:
  - 今日のお題の配信
  - スレッドへの回答通知
  - ピン止め・Award通知
  - フォロー中ユーザーの新規投稿

### 3.8 今日のお題（通知導線）

おすすめテーマの通知導線として機能する。

**要件**:
- 運営がお題を配信（手動）
- 配信時にプッシュ通知を送信
- お題のフォーマット: タイトル + 補足テキスト（任意）
- 回答フォーマットはスレッドと同じ（好きなもの3つの並列リスト）
- 1ユーザー1お題につき1回答
- 回答はプロフィールにも蓄積
- 過去のお題はアーカイブとして閲覧可能
- UI上の配置: ホーム画面上部またはタブとして目立つ位置
- 未回答のお題がある場合はバッジ等で通知

## 7. 非機能要件

### 7.1 セキュリティ

| 項目 | 対応 |
|------|------|
| 認証 | Supabase Auth（Google OAuth）。PKCE対応 |
| 通信 | HTTPS前提 |
| 入力バリデーション | Zod v4スキーマ（API routes） |
| XSS防止 | Reactデフォルトエスケープ。`dangerouslySetInnerHTML`禁止 |
| CSRF | Next.js App RouterのSameSite cookieで自動保護 |
| SQLインジェクション | Supabase REST APIのパラメータ化クエリ |
| NGワード | 投稿・タグ作成時に禁止ワード検査 |
| ブロック・ミュート | 実装済み |

### 7.2 プライバシー

- ユーザーは匿名（ニックネーム + アバター）
- 検索行動のログは運営分析用に保持するが、個別ユーザーに紐づけた外部提供は行わない

### 7.3 パフォーマンス

| 項目 | 目標 |
|------|------|
| 投稿時レコメンドの生成 | 2秒以内 |
| APIレスポンス | Upstash Redisによるレート制限 |
| インプレッション記録 | 一括記録（バッチ処理） |

### 7.4 可用性・運用

- 無料枠内でデータ量・コストを監視
- リリース後は利用状況を継続モニタリング

### 7.5 拡張性

- フェーズ追加を見据え、タグ粒度・公開設定・投稿フォーマットを拡張可能な設計
- マイルストーン・分析メニューの追加は運営側で柔軟に変更可能な設計
